  "name": "My Synth Patch",

    {

      "type": "Filter",

    {
      "id": "out1",

      "to": {"module": "out1", "input": "inR"}
    }
  ]
}

Module Processing Flow ​

Setup Phase (once at boot) ​

  1. Load patch.json from SPIFFS
  2. Parse JSON using ArduinoJson library
  3. For each module in JSON:
    • Create module instance via ModuleFactory
    • Allocate audio buffers (leftBuffer, rightBuffer)
    • Initialize module state from params
  4. Build connection map (which modules feed which)
  5. Topologically sort modules (process order)

Audio Loop Phase (every 1.45ms @ 44.1kHz) ​

cpp
void audioLoop() {
  // Clear all module output buffers
  for (auto& module : modules) {
    memset(module.leftBuffer, 0, 64 * sizeof(float));
    memset(module.rightBuffer, 0, 64 * sizeof(float));
  }

  // Process modules in topological order
  for (auto& module : sortedModules) {
    // Get input buffers from connected modules
    float* leftIn = getConnectedInput(module, "inL");
    float* rightIn = getConnectedInput(module, "inR");

    // Process this module
    module.process(leftIn, rightIn,
                   module.leftBuffer, module.rightBuffer,
                   64);
  }

  // Output final audio to I2S
  i2s_write(I2S_NUM_0, finalLeftBuffer, finalRightBuffer, ...);
}

Memory Management ​

Static Allocation (Known at Compile Time) ​

  • I2S buffers: 2 × 64 samples × 4 bytes = 512 bytes
  • Max module count: 50 modules (configurable)
  • Max connections: 200 connections (configurable)

Dynamic Allocation (Based on Patch) ​

  • Module instances: ~200-500 bytes each
  • Audio buffers: 64 samples × 4 bytes × 2 channels = 512 bytes per module
  • Module state: Varies by type (oscillator phase, filter coefficients, etc.)

Typical Memory Usage ​

  • Simple patch (5 modules): ~5KB
  • Medium patch (15 modules): ~15KB
  • Complex patch (30 modules): ~30KB

ESP32-S3 has 512KB SRAM, so plenty of headroom!

Performance Considerations ​

CPU Budget @ 44.1kHz, 64-sample blocks ​

  • Block period: 64 / 44100 = 1.45ms
  • Available CPU time: 1.45ms × 240MHz = 348,000 cycles
  • Target: <50% CPU usage for stability

Typical Module Costs (Estimated) ​

  • Oscillator: ~2,000 cycles (sine lookup + phase increment)
  • Filter (biquad): ~3,000 cycles (5 multiplies, 4 adds per sample × 64)
  • VCA: ~500 cycles (1 multiply per sample × 64)
  • ADSR: ~1,000 cycles (envelope calculation + multiply)
  • Reverb (simple): ~15,000 cycles (feedback delays + mixing)

Optimization Strategies ​

  1. Reuse codegen DSP code - already optimized
  2. Minimize allocations - pre-allocate buffers
  3. Cache module params - don't parse JSON every loop
  4. Use PSRAM for samples - keep SRAM for processing
  5. Profile and optimize hot paths - use ESP32 profiler

Design Decisions ​

Why Function Dispatch Over Virtual Functions? ​

  • Simpler to debug - clear if/else chain
  • Easier to add modules - just add case to switch
  • Overhead acceptable - ~10-20% vs virtual functions at ~5-10%
  • Can optimize later - profile first, optimize if needed

Why SPIFFS Over SD for Active Patch? ​

  • Faster access - no SD card latency
  • Reliable - SD cards fail, SPIFFS is solid state
  • Hybrid approach - active patch in SPIFFS, library on SD

Why 64-Sample Blocks? ​

  • Matches codegen - same latency characteristics
  • Low latency - 1.45ms is imperceptible
  • Efficient - good balance between overhead and latency

Error Handling ​

Invalid JSON ​

  • Log error to serial
  • Fall back to default "safe" patch (simple oscillator)
  • Flash LED pattern to indicate error state

Module Creation Failure ​

  • Skip module, log warning
  • Continue processing other modules
  • Avoid crash at all costs (workshop environment!)

Buffer Overflow ​

  • Clip audio to ±1.0 range
  • Log warning (don't spam serial)
  • Consider reducing module count in future uploads

Future Enhancements ​

Phase 2: Live Parameter Updates ​

  • Parse serial commands for parameter changes
  • Update module params without stopping audio
  • Smooth transitions (no clicks/pops)

Phase 3: Module Hot-Swapping ​

  • Add/remove modules without audio interruption
  • Crossfade between old and new configurations
  • Requires double-buffering and careful synchronization

Phase 4: Multi-Patch Management ​

  • Bank of patches on SD card
  • Switch via encoder/buttons
  • Preset management system

Development Guidelines ​

  1. Test with real patches - don't just unit test modules
  2. Profile early - measure CPU usage from day one
  3. Keep codegen working - never break the fallback system
  4. Document everything - future you will thank present you
  5. Incremental development - get one module working perfectly before adding next
