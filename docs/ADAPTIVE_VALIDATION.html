Adaptive Validation System ​

Core Principle ​

"Guide, don't dictate. Warn, don't block. Learn from reality, not assumptions."

Users know their creative intent better than we do. The system should:

  1. Provide information (CPU estimates, warnings)
  2. Learn from actual hardware behavior
  3. Only hard-block truly catastrophic scenarios
  4. Let users make informed decisions

Three Validation Tiers ​

Tier 1: INFORMATION (Always Show, Never Block) ​

Real-time budget display showing estimated resource usage:

┌─────────────────────────────────────────┐
│ Patch Resources (Estimated)             │
├─────────────────────────────────────────┤
│ CPU:   [████████████░] 83%  ⚠ High     │
│ PSRAM: [███░░░░░░░░░░] 21%  ✓ OK       │
│                                          │
│ ℹ️  Estimates may vary with actual use  │
└─────────────────────────────────────────┘

Purpose: Educate user about resource usage Action: None - just information

Tier 2: WARNINGS (Advise, Suggest, Allow) ​

Soft warnings for patches that might have issues:

┌─────────────────────────────────────────────────┐
│ ⚠ High Resource Usage Detected                 │
├─────────────────────────────────────────────────┤
│ Estimated CPU: 94%                              │
│                                                  │
│ This patch may experience:                      │
│  • Brief audio glitches during heavy sections   │
│  • Clicks/pops when triggering all modules      │
│                                                  │
│ ℹ️  Many patches work fine above estimates!     │
│                                                  │
│ What would you like to do?                      │
│                                                  │
│ [ Simplify Patch ]  [ Upload Anyway ] [ Cancel ]│
└─────────────────────────────────────────────────┘

Triggered by:

  • Estimated CPU 80-120%
  • PSRAM 60-90%
  • 2+ expensive modules (Reverb + Granular)
  • Circular connections (potential feedback)
  • Missing connections to output

User choice: Proceed or reconsider Default action: Allow upload

"Upload Anyway" teaches the system:

  userRating: 'no_issues'
});

Tier 3: HARD BLOCKS (Only Truly Dangerous Scenarios) ​

Block uploads only when:

  1. Known crash scenarios (discovered through testing):

    javascript
    const KNOWN_CRASH_PATTERNS = [
  {
    modules: { 'Reverb': 3 },  // 3 reverbs = confirmed crash
    reason: 'PSRAM exhaustion causes heap corruption'
  },
  {
    modules: { 'Granular': 2, 'BPMLooper': 2 },
    reason: 'Simultaneous PSRAM access deadlock'
  }
];

  2. Physical limits (cannot be exceeded):

    • Total PSRAM allocation > 7.5MB (hard limit)
    • Sample files larger than available PSRAM
    • Invalid JSON structure

  3. Logic errors (mathematically impossible):

    • Circular dependencies that create infinite loops
    • Connections to non-existent modules
    • Missing required connections (e.g., no output module)

Example hard block:

┌─────────────────────────────────────────────────┐
│ ❌ Cannot Upload - Hardware Limitation          │
├─────────────────────────────────────────────────┤
│ This patch uses 8.2MB PSRAM                     │
│ ESP32-S3 has 7.5MB available                    │
│                                                  │
│ This WILL crash the device.                     │
│                                                  │
│ Reduce:                                          │
│  • Sample sizes (currently 5MB)                 │
│  • Looper length (currently 15 sec)             │
│  • Granular buffer (currently 2 sec)            │
│                                                  │
│ [ Fix Patch ]                    [ Cancel ]     │
└─────────────────────────────────────────────────┘

No "Upload Anyway" option - this is a physical impossibility.

Dynamic Threshold Adjustment ​

javascript
class AdaptiveBudget {
  constructor() {
    this.cpuWarningThreshold = 85;  // Initial conservative value
    this.adjustmentHistory = [];
  }

  learn(estimate, actual, result) {
    this.adjustmentHistory.push({ estimate, actual, result });

    // If 10 patches with 90-100% estimates worked fine, raise threshold
    const recentHighCPU = this.adjustmentHistory
      .slice(-20)
      .filter(h => h.estimate > 90 && h.result === 'stable');

    if (recentHighCPU.length > 10) {
      this.cpuWarningThreshold = Math.min(110, this.cpuWarningThreshold + 5);
      console.log(`Raised CPU warning threshold to ${this.cpuWarningThreshold}%`);
    }

    // If patches under threshold are glitching, lower it
    const recentGlitches = this.adjustmentHistory
      .slice(-20)
      .filter(h => h.estimate < this.cpuWarningThreshold && h.result === 'glitchy');

    if (recentGlitches.length > 3) {
      this.cpuWarningThreshold = Math.max(70, this.cpuWarningThreshold - 5);
      console.log(`Lowered CPU warning threshold to ${this.cpuWarningThreshold}%`);
    }
  }

  shouldWarn(estimatedCPU) {
    return estimatedCPU > this.cpuWarningThreshold;
  }
}

Smart Warning System ​

Context-Aware Warnings ​

Don't just check totals - understand the patch:

javascript
function intelligentValidation(modules, connections) {
  const warnings = [];

  // Pattern: Reverb + Granular together
  if (hasModule('Reverb') && hasModule('Granular')) {
    // Check if they're in series (more expensive) or parallel
    const inSeries = isDownstream('Reverb', 'Granular') ||
                     isDownstream('Granular', 'Reverb');

    if (inSeries) {
      warnings.push({
        level: 'HIGH',
        message: 'Reverb → Granular in series is CPU-intensive',
        suggestion: 'Consider parallel routing or shorter reverb decay',
        allowOverride: true
      });
    } else {
      warnings.push({
        level: 'MEDIUM',
        message: 'Reverb + Granular uses significant resources',
        suggestion: 'Monitor for audio glitches',
        allowOverride: true
      });
    }
  }

  // Pattern: Oscillator with no ADSR (will drone constantly)
  const oscWithoutEnvelope = modules
    .filter(m => m.type === 'Oscillator')
    .filter(osc => !hasEnvelopeControl(osc));

  if (oscWithoutEnvelope.length > 2) {
    warnings.push({
      level: 'INFO',
      message: `${oscWithoutEnvelope.length} oscillators have no volume control`,
      suggestion: 'Consider adding VCA + ADSR for dynamics',
      allowOverride: true  // Not wrong, just unusual
    });
  }

  // Pattern: Disconnected modules (dead code)
  const disconnected = findDisconnectedModules(modules, connections);
  if (disconnected.length > 0) {
    warnings.push({
      level: 'INFO',
      message: `${disconnected.length} modules not connected to output`,
      suggestion: 'Remove unused modules to free resources',
      allowOverride: true
    });
  }

  return warnings;
}

ESP32-Side Intelligence ​

Real-Time Performance Monitoring ​

cpp
class PerformanceMonitor {
  public:
    struct Stats {
      float avgCPU;
      float peakCPU;
      int glitchCount;
      unsigned long uptime;
    };

    void update(unsigned long blockTime) {
      float cpuPercent = (blockTime / 1450.0f) * 100;

      // Exponential moving average
      avgCPU = (avgCPU * 0.95f) + (cpuPercent * 0.05f);
      peakCPU = max(peakCPU, cpuPercent);

      if (blockTime > 1450) {
        glitchCount++;
      }

      // Reset peak every 10 seconds
      if (millis() - lastReset > 10000) {
        peakCPU = avgCPU;
        lastReset = millis();
      }
    }

    Stats getStats() {
      return { avgCPU, peakCPU, glitchCount, millis() };
    }

    bool isStable() {
      return avgCPU < 90 && glitchCount < 5;
    }
};

Adaptive Module Quality ​

If CPU usage is high, automatically reduce quality:

cpp
void processAudioBlock() {
  unsigned long start = micros();

  // Normal processing
  for (auto* module : modules) {
    processModule(module);
  }

  unsigned long elapsed = micros() - start;

  // Auto-adjust quality if consistently over budget
  if (elapsed > 1450) {
    adaptiveQuality.reduce();  // Lower reverb tail, reduce filter order, etc.
  } else if (elapsed < 1000) {
    adaptiveQuality.increase();  // Restore quality if we have headroom
  }
}

class AdaptiveQuality {
  void reduce() {
    // Reduce reverb tail length
    for (auto* module : modules) {
      if (module->type == MOD_REVERB) {
        float currentDecay = module->params["roomSize"];
        module->params["roomSize"] = currentDecay * 0.8f;  // 20% shorter
      }
    }

    Serial.println("Reduced quality to maintain stability");
  }
};

Validation API Design ​

Flexible Validation Function ​

javascript
// GenRuntime/web-app/src/validation/PatchValidator.js

class PatchValidator {
  constructor(options = {}) {
    this.strictness = options.strictness || 'balanced';  // 'protective', 'balanced', 'permissive'
    this.learningEnabled = options.learningEnabled !== false;
    this.adaptiveBudget = new AdaptiveBudget();
  }

  validate(patch) {
    const result = {
      canUpload: true,  // Innocent until proven guilty
      hardBlocks: [],   // Cannot proceed
      warnings: [],     // Proceed with caution
      info: [],         // FYI
      estimates: {}
    };

    // Always calculate estimates (Tier 1: INFORMATION)
    result.estimates = {
      cpu: this.estimateCPU(patch.modules),
      psram: this.estimatePSRAM(patch.modules)
    };

    // Check hard limits (Tier 3: HARD BLOCKS)
    const hardLimitChecks = [
      this.checkPSRAMLimit(patch),
      this.checkKnownCrashPatterns(patch),
      this.checkLogicErrors(patch)
    ];

    result.hardBlocks = hardLimitChecks.filter(b => b !== null);

    if (result.hardBlocks.length > 0) {
      result.canUpload = false;
      return result;
    }

    // Soft warnings (Tier 2: WARNINGS)
    if (this.strictness !== 'permissive') {
      const threshold = this.adaptiveBudget.cpuWarningThreshold;

      if (result.estimates.cpu > threshold) {
        result.warnings.push({
          type: 'HIGH_CPU',
          severity: result.estimates.cpu > 120 ? 'high' : 'medium',
          message: `Estimated CPU ${result.estimates.cpu}% exceeds typical usage`,
          suggestion: 'Consider simplifying patch or upload anyway to test',
          canOverride: true
        });
      }
    }

    // Context-aware warnings
    result.warnings.push(...this.intelligentWarnings(patch));

    return result;
  }

  // Learn from actual hardware feedback
  reportFeedback(patchId, actualPerformance) {
    if (!this.learningEnabled) return;

    const patch = this.patchHistory.get(patchId);
    if (!patch) return;

    this.adaptiveBudget.learn(
      patch.estimates.cpu,
      actualPerformance.avgCPU,
      actualPerformance.stability
    );

    console.log('Learned from patch:', {
      estimated: patch.estimates.cpu,
      actual: actualPerformance.avgCPU,
      result: actualPerformance.stability
    });
  }
}

export default PatchValidator;

Usage in Upload Flow ​

javascript
// In your upload button handler

async function handleUpload() {
  const validator = new PatchValidator({
    strictness: settings.validationStrictness,
    learningEnabled: true
  });

  const patch = {
    modules: modules,
    connections: connections
  };

  const validation = validator.validate(patch);

  // Always show information
  console.log('Patch estimates:', validation.estimates);

  // Hard blocks - cannot proceed
  if (!validation.canUpload) {
    showErrorDialog(validation.hardBlocks);
    return;
  }

  // Warnings - can proceed with confirmation
  if (validation.warnings.length > 0) {
    const userChoice = await showWarningDialog(validation.warnings);

    if (userChoice === 'cancel') return;
    if (userChoice === 'simplify') {
      // Help user reduce patch complexity
      showSimplificationSuggestions();
      return;
    }
    // userChoice === 'proceed' → continue to upload
  }

  // Upload patch
  const patchId = await uploadToESP32(patch);

  // After upload, monitor performance
  setTimeout(async () => {
    const performance = await getESP32Performance();
    validator.reportFeedback(patchId, performance);

    if (!performance.stable) {
      showNotification('Patch is unstable - consider simplifying');
    }
  }, 5000);  // Check after 5 seconds
}

Summary: The Balanced Approach ​

What we DO:

  • ✅ Show real-time resource estimates
  • ✅ Warn about potentially problematic patterns
  • ✅ Block physically impossible patches (PSRAM overflow)
  • ✅ Learn from actual hardware behavior
  • ✅ Adapt thresholds based on experience
  • ✅ Provide "Upload Anyway" for warnings
  • ✅ Auto-degrade quality before crashing

What we DON'T DO:

  • ❌ Block patches based on estimates alone
  • ❌ Assume our calculations are perfect
  • ❌ Prevent creative experimentation
  • ❌ Hard-code arbitrary limits
  • ❌ Crash the ESP32 without recovery

Result:

  • Students can experiment freely
  • System provides helpful guidance
  • Real crashes are prevented (hardware watchdog + safe fallback)
  • Over time, system learns what actually works
  • Creative patches that "shouldn't work" but do are allowed
