Dynamic CPU Frequency Adjustment ​

Overview ​

GenRuntime includes an adaptive CPU frequency system that automatically adjusts the ESP32's clock speed based on real-time audio processing load. This provides optimal performance while minimizing power consumption and heat generation.

Benefits ​

For Complex Patches ​

  • Automatically increases clock speed when CPU usage is high
  • Prevents audio glitches without manual intervention
  • Allows running heavier patches (Reverb + Granular + multiple oscillators)

For Simple Patches ​

  • Automatically reduces clock speed when CPU usage is low
  • Saves power (important for battery/USB-powered systems)
  • Reduces heat generation
  • Extends hardware lifespan

For Workshops ​

  • Students don't need to understand CPU frequencies
  • "Auto" mode just works
  • No manual tweaking required
  • Better experience on underpowered USB ports

How It Works ​

Performance Monitoring ​

Every audio block (64 samples @ 44.1kHz = 1.45ms):

  1. Measure time taken to process the block
  2. Calculate CPU usage percentage
  3. Track average usage (exponential moving average)
  4. Track peak usage (last 10 seconds)
  5. Detect glitches (blocks that exceeded 1.45ms)

Adaptive Decision Making ​

Every 5 seconds:

  1. Evaluate current CPU load (average and peak)
  2. Determine optimal frequency for that load
  3. If frequency change needed, apply it
  4. Test stability for 1 second
  5. Revert if instability detected

Frequency Selection Logic ​

AUTO Mode:

Peak CPU < 40%  → 240MHz (plenty of headroom, save power)
Peak CPU 40-50% → Stay at current speed (stable)
Peak CPU 50-70% → Current speed is optimal
Peak CPU 70-85% → Increase 40MHz (need more headroom)
Peak CPU > 85%  → Increase 40MHz (getting close to limits)

POWERSAVE Mode:

Peak CPU < 50%  → 240MHz (maximize battery life)
Peak CPU 50-85% → 240MHz (tolerate higher usage for power savings)
Peak CPU > 85%  → 280MHz only (minimal increase)

Clock Modes ​

FIXED Mode (Default) ​

  • Runs at user-specified frequency always
  • Predictable, consistent behavior
  • Good for testing/benchmarking
  • Use when you know your patch's requirements

Use cases:

  • Testing patch at specific speed
  • Benchmarking DSP performance
  • Patches with known stable speed
  • When you want full control

Example:

json
{
  "clockSpeed": 320,
  "clockMode": 0,
  ...
}

AUTO Mode (Recommended for Most Users) ​

  • Automatically adjusts based on load
  • Starts at specified clockSpeed as baseline
  • Increases when needed, decreases when possible
  • Balances performance and power

Use cases:

  • Patches with variable complexity (sequenced)
  • General-purpose patches
  • Workshop/educational use
  • When patch complexity is unknown

Example:

json
{
  "clockSpeed": 240,
  "clockMode": 1,
  ...
}

Behavior:

Patch starts → 240MHz
5 seconds later → CPU at 25%, stay at 240MHz
User triggers complex section → CPU peaks at 88%
5 seconds later → System detects high load, increases to 280MHz
Complex section → Now running at 65% CPU ✓
Section ends → CPU drops to 40%
10 seconds later → System reduces to 240MHz to save power

POWERSAVE Mode ​

  • Prioritizes low power over performance
  • Only increases speed when absolutely necessary
  • Good for battery-powered applications
  • May allow occasional glitches to save power

Use cases:

  • Battery-powered installations
  • Low-power embedded systems
  • Patches that can tolerate occasional glitches
  • When power consumption is critical

Example:

json
{
  "clockSpeed": 240,
  "clockMode": 2,
  ...
}

Hardware Considerations ​

v1.0 PCB (Current Hardware) ​

With Original Polyfuse (1206L075 - 0.75A):

  • 240MHz: ✓ Safe
  • 280MHz: ⚠️ May brownout after 10+ minutes
  • 320MHz: ✗ Will brownout

Recommended: Bypass polyfuse or replace with 1.5A version

With Bypassed Polyfuse:

  • 240MHz: ✓ Safe
  • 280MHz: ✓ Safe
  • 320MHz: ✓ Safe (monitor temperature)
  • 360MHz: ⚠️ Test stability
  • 400MHz: ⚠️ May be unstable

Power consumption:

  • 240MHz: ~350mA @ 3.3V = 1.15W
  • 280MHz: ~400mA @ 3.3V = 1.32W
  • 320MHz: ~450mA @ 3.3V = 1.49W
  • 360MHz: ~500mA @ 3.3V = 1.65W

TLV75733 LDO dissipation:

  • 240MHz: 0.60W (safe)
  • 280MHz: 0.68W (warm)
  • 320MHz: 0.77W (hot but should work)
  • 360MHz: 0.85W (very hot, may thermal shutdown)

v1.1 PCB (Future Hardware) ​

With Buck Converter + 1.5A Polyfuse:

  • All speeds 240-400MHz: ✓ Fully supported
  • No thermal issues
  • Full AUTO mode capability

Best Practices ​

For Patch Design ​

1. Test with AUTO mode first:

json
{
  "clockSpeed": 240,
  "clockMode": 1
}

Let the system find optimal speed automatically.

2. If stable, note the frequency used:

Serial> PERF
CPU Frequency: 280 MHz  ← System chose this
CPU Usage (avg): 65.0%

3. For production/performance, switch to FIXED:

json
{
  "clockSpeed": 280,
  "clockMode": 0
}

Ensures consistent behavior.

For Development ​

Enable verbose logging:

Serial> AUTO

Adaptive: Adjusting 240 → 280 MHz (CPU: 78.5%, Peak: 85.2%)
CPU frequency changed: 240 → 280 MHz

[PERF] CPU: avg=65.3% peak=78.1% freq=280MHz glitches=0
(Every 30 seconds)

Monitor for issues:

  • Watch for glitches (should be 0 in stable patches)
  • Check if frequency bounces up/down (indicates borderline load)
  • Verify audio quality at different speeds

For Workshops ​

Recommend AUTO mode for students:

javascript
// In web app, default new patches to:
{
  "clockSpeed": 240,
  "clockMode": 1  // AUTO
}

Advantages:

  • Students don't think about CPU speed
  • Patches adapt to their complexity
  • Better success rate on varying hardware
  • Teaches good practices (system optimization)

Tuning Parameters ​

If you want to adjust the adaptive behavior, edit PerformanceMonitor.h:

Thresholds ​

cpp
// In calculateOptimalFrequency():

// Increase speed threshold
if (load > 85.0f) {  // Change to 80.0f or 90.0f
  // Need more headroom
}

// Decrease speed threshold
if (load < 40.0f) {  // Change to 30.0f or 50.0f
  // Plenty of headroom
}

Adjustment Frequency ​

cpp
// In loop():

// How often to check (default: every 5 seconds)
if (millis() - lastAdaptiveCheck > 5000) {  // Change to 3000 or 10000
  perfMon.adaptiveAdjust();
}

Frequency Steps ​

cpp
// In calculateOptimalFrequency():

if (currentFreq < FREQ_320) {
  return currentFreq + 40;  // Change to +80 for bigger jumps
}

Troubleshooting ​

Patch is glitching at 240MHz ​

Symptom: Audio clicks/pops, glitch count increasing

Solution:

Serial> CLOCK 280
(or)
Serial> AUTO  (let system handle it)

Frequency keeps bouncing between 240-280MHz ​

Symptom: Every 5 seconds frequency changes

Cause: Patch is right on the edge of 240MHz capacity

Solutions:

  1. Lock to 280MHz: CLOCK 280
  2. Simplify patch slightly
  3. Adjust threshold (see Tuning Parameters above)

ESP32 resets unexpectedly ​

Symptom: Runtime restarts after a few minutes at high speed

Cause: Likely power supply issue (polyfuse thermal runaway)

Solutions:

  1. Bypass polyfuse temporarily (for testing)
  2. Use POWERSAVE mode: POWERSAVE
  3. Limit max speed in code (edit FREQ constants)
  4. Use better USB power supply (2A minimum)

AUTO mode never increases speed ​

Symptom: Stays at 240MHz even when glitching

Cause: Peak CPU not being detected properly

Debug:

Serial> PERF
(Check peak CPU - should reflect worst-case load)

If peak is low but glitches occur:
- May need to adjust averaging (95% old, 5% new is conservative)
- Try changing to 90% old, 10% new for faster response

Performance Metrics ​

Typical Patch Examples ​

Simple (1 Osc → Out):

  • CPU: ~15% @ 240MHz
  • AUTO mode: Stays at 240MHz
  • Power: Minimal

Medium (2 Osc → Filter → VCA → Out):

  • CPU: ~45% @ 240MHz
  • AUTO mode: Stays at 240MHz
  • Power: Low

Complex (Reverb + Granular + 3 Osc):

  • CPU: ~95% @ 240MHz (glitches!)
  • AUTO mode: Increases to 320MHz → 70% CPU ✓
  • Power: Moderate

Extreme (2 Reverb + Granular + Delays):

  • CPU: Would exceed 100% @ 240MHz
  • AUTO mode: Increases to 360-400MHz (if hardware supports)
  • Power: High
  • Note: Validate this doesn't exceed PSRAM bandwidth!
